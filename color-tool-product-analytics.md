# Продуктовая аналитика: Генератор UI-палитр на базе OKLCH

## 1. Проблема и контекст

### Текущий workflow дизайнера (болевой путь)

Сейчас создание цветовой палитры для нового проекта — это цепочка из 4–6 ручных шагов по разным инструментам, каждый из которых решает только часть задачи:

**Шаг 1 → A Nice Red** — получение семантических цветов (success, warning, danger) + нейтрального оттенка по бренд-цвету.
- ✅ Быстро подбирает гармоничные статусные цвета
- ❌ Не генерирует info (синий оттенок)
- ❌ Не учитывает OKLCH / перцептуальную модель
- ❌ Нет проверки контрастности

**Шаг 2 → Radix UI Custom Palette** — прогон каждого цвета через генератор 12-шаговой шкалы.
- ✅ Семантически выверенная 12-шаговая система + alpha-канал
- ✅ Поддержка light/dark тем
- ❌ Нужно прогонять каждый цвет отдельно (циклично: brand → success → warning → danger → info → neutral)
- ❌ Интерполяция к ближайшей готовой палитре — шаг 9 не равен входящему HEX
- ❌ Нет коррекции яркости/насыщенности между темами для баланса восприятия
- ❌ Проблемы с брендбуком: если бренд-цвет фиксирован, шаги вокруг него могут "сломаться"

**Шаг 3 → Ручная коррекция** — подгон под брендбук, проверка контрастности, финальные правки.

### Ключевые боли

| Боль | Суть | Влияние |
|------|------|---------|
| **Фрагментация** | 3+ инструмента для одной задачи | Потеря времени, ошибки переноса |
| **Цикличность** | Прогон каждого оттенка отдельно | 5–7 итераций вместо одной |
| **Неточность бренд-цвета** | Radix интерполирует к ближайшему, шаг 9 ≠ входящий HEX | Конфликт с утвержденным брендбуком |
| **Отсутствие info** | A Nice Red не генерит синий | Ручной подбор |
| **Нет темовой коррекции** | dark/light темы без учета восприятия | Визуальный дисбаланс |
| **Негибкость** | Нет режимов для разных сценариев (пастельный бренд, максимальная хроматика, фиксированный брендбук) | Один алгоритм не подходит всем |
| **Нет APCA** | Большинство инструментов проверяют только WCAG 2.x | Устаревшие данные о читаемости |

## 2. Продуктовая идея

**Единый генератор комплексных UI-палитр**, который:
- Принимает один бренд-цвет (HEX) и за одно действие генерирует полный набор: brand, success, warning, danger, info, neutral — по 12 шагов каждый + alpha-варианты
- Работает в OKLCH-пространстве для перцептуальной точности
- Поддерживает light/dark темы с коррекцией яркости/хроматики
- Даёт гибкие режимы для нестандартных сценариев
- Проверяет контрастность по APCA и WCAG 2.x
- Экспортирует в CSS variables, Figma, Tailwind, JSON

## 3. Архитектура модулей

### 3.1. Модуль входных данных (Input Layer)

**Входные параметры:**
- `brandColor: HEX` — основной цвет бренда
- `mode: "auto" | "fixed-brand" | "max-chroma" | "equal-chroma"` — режим генерации
- `theme: "light" | "dark" | "both"` — целевая тема
- `neutralStyle: "pure-gray" | "tinted"` — стиль нейтралов
- `contrastStandard: "APCA" | "WCAG2" | "both"` — стандарт контрастности
- `gamut: "sRGB" | "P3"` — целевой гамут
- `customOverrides: { [step]: HEX }` — ручные переопределения конкретных шагов (для брендбуков)

**Конвертация:** На входе HEX/RGB → мгновенная конвертация в OKLCH(L, C, H) через Oklab. Все дальнейшие вычисления — в OKLCH.

### 3.2. Модуль подбора семантических оттенков (Semantic Hue Resolver)

По входящему бренд-цвету определяются гармоничные hue-углы для семантических ролей.

**Логика подбора оттенков:**

```
Brand HEX → OKLCH(L, C, H_brand)

Success:  H ≈ 145° (зелёный диапазон 130–160°)
Warning:  H ≈ 80°  (жёлто-оранжевый 60–90°)  
Danger:   H ≈ 25°  (красный диапазон 15–35°)
Info:     H ≈ 245° (синий диапазон 230–260°)
Neutral:  H = H_brand, C ≈ 0.01–0.02 (tinted) или C = 0 (pure gray)
```

**Гармонизация с брендом:**
Оттенки не фиксированные — они сдвигаются в сторону бренд-цвета для гармоничности. Если бренд-цвет — синий с H=250°, то info может получить H=245° (ближе к бренду), а success немного сдвинуться к H=150° (дальше от синего, для контраста). Алгоритм: чем ближе hue семантического цвета к бренду — тем сильнее отодвигаем, чтобы избежать слияния.

Это то, что делает A Nice Red — но в OKLCH и с добавлением info.

### 3.3. Модуль генерации 12-шаговой шкалы (Scale Generator)

Ядро системы. Для каждого цвета (brand, success, warning, danger, info, neutral) генерируются 12 шагов по модели Radix.

**Семантика шагов (по Radix UI):**

| Шаг | Назначение | L (light) | L (dark) |
|-----|-----------|-----------|----------|
| 1 | App background | ~0.99 | ~0.13 |
| 2 | Subtle background | ~0.97 | ~0.16 |
| 3 | UI element bg | ~0.94 | ~0.20 |
| 4 | Hovered UI bg | ~0.91 | ~0.24 |
| 5 | Active/Selected bg | ~0.87 | ~0.28 |
| 6 | Subtle border | ~0.83 | ~0.33 |
| 7 | UI border, focus ring | ~0.76 | ~0.39 |
| 8 | Strong border, hover | ~0.65 | ~0.47 |
| 9 | Solid background (основной) | ~0.55 | ~0.55 |
| 10 | Hovered solid bg | ~0.50 | ~0.60 |
| 11 | Low-contrast text | ~0.40 | ~0.75 |
| 12 | High-contrast text | ~0.25 | ~0.88 |

**Алгоритм генерации (OKLCH):**

Для каждого шага определяется тройка (L, C, H):
- **H (hue):** фиксирован от Semantic Hue Resolver (с незначительной вариацией для натуральности)
- **L (lightness):** по кривой от ~0.99 (шаг 1) до ~0.25 (шаг 12) в light theme. В dark theme — инвертированная кривая
- **C (chroma):** по кривой с пиком на шаге 9 (максимальная насыщенность)

**Критически важное отличие от Radix:** шаг 9 — это ТОЧНО входящий цвет пользователя (или его OKLCH-hue), а не интерполяция к ближайшей готовой палитре. Остальные шаги строятся от этой точки.

### 3.4. Модуль режимов хроматики (Chroma Strategy Engine)

Это решение проблемы "жёлтый vs синий" и пастельных брендов.

**Режим 1: `equal-chroma` (равная хроматика)**
Все оттенки (success, warning, danger, info) генерируются с одинаковым пиком хроматики. Подбирается минимальный "потолок" C, при котором ВСЕ оттенки проходят по гамуту (не выходят за sRGB/P3).

Зачем: жёлтый в OKLCH имеет пик хроматики ~0.21, а синий — ~0.31. Если не ограничивать, синий будет визуально "ярче". Equal chroma уравнивает восприятие.

**Режим 2: `max-chroma` (максимальная насыщенность)**
Каждый оттенок получает максимально возможную хроматику в рамках гамута. Жёлтый будет C=0.21, синий — C=0.31. Цвета максимально насыщенные, но визуально неравномерные.

Зачем: для маркетинговых лендингов, промо-материалов, где нужна максимальная яркость каждого цвета.

**Режим 3: `fixed-brand` (фиксированный бренд)**
Шаг 9 (и опционально шаг 10) жёстко привязан к HEX из брендбука. Остальные шаги строятся от этой фиксированной точки, с допустимыми отклонениями L и C.

Зачем: когда брендбук утвержден и цвет менять нельзя, даже если он пастельный или необычный.

**Режим 4: `auto` (по умолчанию)**
Комбинация: brand-цвет фиксирован на шаге 9, семантические цвета генерируются с equal-chroma, нейтралы — tinted gray.

### 3.5. Модуль темовой адаптации (Theme Adapter)

Генерация dark-темы — не просто инверсия lightness. Нужна коррекция:

**Коррекция хроматики для dark theme:**
- В тёмном режиме насыщенные цвета на тёмном фоне воспринимаются ярче (эффект Хелмгольца-Кольрауша). Нужно снижать C на 10–20% для шагов 9–10
- Шаги 1–5 (фоновые) в dark theme должны иметь минимальную хроматику, чтобы не утомлять глаз
- Шаги 11–12 (текстовые) нуждаются в повышенной L для читаемости на тёмном фоне

**Коррекция lightness для dark theme:**
- Зеркальная инверсия L (light_step_1 → dark_step_12) недостаточна — нужна нелинейная кривая, учитывающая адаптацию глаза к тёмному интерфейсу
- Контраст между шагами в dark theme должен быть чуть выше, чем в light (глаз менее чувствителен к контрастам на тёмном фоне)

### 3.6. Модуль гамут-маппинга (Gamut Mapper)

Не все OKLCH-значения представимы в sRGB. Модуль отвечает за "приземление" цветов:

**Стратегии маппинга:**
- **Reduce chroma first (рекомендуемая CSSWG):** снижаем C, сохраняя L и H. Цвет тускнеет, но остаётся того же оттенка и яркости
- **Reduce lightness second:** если после снижения C до 0 цвет всё ещё вне гамута — корректируем L
- **P3 fallback:** для P3-устройств можно оставить более насыщенные цвета, с sRGB-fallback

**Обнаружение проблемных зон:**
Некоторые комбинации H и C при высокой L гарантированно вне sRGB (например, ярко-жёлтый при L=0.95 или насыщенный синий при L=0.3). Модуль заранее предупреждает пользователя и предлагает коррекцию.

### 3.7. Модуль контрастности (Contrast Checker)

Встроенная проверка всех цветовых пар.

**APCA (Accessible Perceptual Contrast Algorithm):**
- Lc 90 — предпочтительный для body text (≥14px/400)
- Lc 75 — минимум для body text (≥18px/400)
- Lc 60 — минимум для крупного текста, заголовков
- Lc 45 — минимум для large text, пиктограмм
- Lc 15 — порог видимости

APCA учитывает полярность (light-on-dark vs dark-on-light), что критично для dark theme.

**WCAG 2.x (для обратной совместимости):**
- 4.5:1 — нормальный текст (AA)
- 3:1 — крупный текст (AA)
- 7:1 — нормальный текст (AAA)

**Автоматические проверки:**
- Шаги 11 и 12 на фоне шага 1 и 2 — читаемость текста
- Шаг 9 с белым/чёрным текстом — определение контрастного foreground
- Шаг 3 на фоне шага 1 — различимость компонентов
- Шаги 6–8 на фоне шага 1 — видимость бордеров

### 3.8. Модуль alpha-каналов (Alpha Generator)

Для каждого шага генерируется alpha-версия — полупрозрачный цвет, который визуально совпадает с solid-версией при наложении на указанный фон. Это то, что делают Radix Alpha scales и инструмент Alphredo (alphredo.app).

**Входные данные:**
- Solid-цвет каждого шага (из Scale Generator)
- Цвет фона (`backgroundColor`) — обычно шаг 1 (app background) или белый/чёрный

**Алгоритм:**
Для solid-цвета `S` на фоне `B` найти `alpha` и `foreground color F`, такие что:
```
blend(F, alpha, B) = S
```
где blend — стандартная alpha-compositing формула:
```
result = F * alpha + B * (1 - alpha)
```

Решается для каждого канала (R, G, B) или в OKLCH. Radix решает это для каждого шага, получая пару (цвет, alpha), визуально идентичную solid-версии на целевом фоне.

**Зависимость от фона:** Alpha-цвета привязаны к конкретному фону. При смене фона (например, карточка на сером вместо белого) — alpha-цвет даёт другой визуальный результат. Поэтому нужен параметр `backgroundColor`, а не фиксированный белый.

**Wide-gamut нюанс (из Radix docs):** Alpha blending работает по-разному в sRGB и P3. Для P3-дисплеев нужны отдельные wide-gamut alpha-определения, чтобы сохранить насыщенность. Radix уже генерирует оба варианта.

**Зачем:** Alpha-цвета корректно работают на разных фонах (полосатые таблицы, карточки на подложке), плавно перекрывают контент, незаменимы для overlay, backdrop, glass-morphism.

### 3.9. Модуль интерполяции кривых (Curve Engine)

**Критически важный момент архитектуры.** Кривые L (lightness) и C (chroma) для 12 шагов — это не произвольная математическая функция. Это дизайн-решение, выверенное под конкретные UI-задачи: шаги 1–2 едва различимы, 3–5 дают чёткий диапазон для interactive states, 6–8 — скачок для бордеров, 9 — пик, 11–12 — текст с достаточным контрастом.

#### Уровень 1: Базовая кривая (Radix-extracted) — ДЕФОЛТ

Radix Colors — палитры, собранные вручную. У каждого оттенка своя кривая, но они достаточно близки. Подход к извлечению базовой кривой:

1. Взять Cyan (или другой "нейтральный" hue, у которого нет аномалий по гамуту) как эталон
2. Сконвертировать все 12 шагов Radix Cyan в OKLCH
3. Зафиксировать значения L₁…L₁₂ и C₁…C₁₂ как **эталонную кривую**
4. Для генерации палитры нового оттенка: подставить новый H, применить эталонные L и C, скорректировать C с учётом гамут-ограничений нового hue

Это даёт результат, максимально близкий к Radix — проверенный, знакомый дизайнерам, с правильной семантикой шагов.

**Дополнительно:** можно усреднить кривые нескольких нейтральных оттенков (Cyan, Blue, Teal) для более робастной базовой кривой, исключив outlier-оттенки вроде Yellow.

**Отдельные кривые для accent и gray:** Radix использует разные алгоритмы для accent-цветов и gray. Это нужно сохранить — gray-шкала имеет минимальную хроматику и другое распределение L (более равномерное, без резкого пика на шаге 9).

#### Уровень 2: Кривые Poline — ПРОДВИНУТЫЙ РЕЖИМ

Poline (https://meodai.github.io/poline/) — библиотека для генерации палитр через полярные координаты с easing-функциями. Она даёт красивую математику для произвольных палитр, но **у неё нет "UI-кривой"** — её sinusoidal, exponential, cubic и другие position functions создают эстетичные градиенты, не привязанные к семантике 12-шаговой системы.

**Где Poline полезна:**
- Как набор **альтернативных пресетов** для дизайнеров, которые хотят нестандартный характер палитры (более контрастную, более мягкую)
- Для **гармоник Иттена** (модуль 3.10), где нет жёсткой семантики шагов и нужны плавные переходы между якорями
- Для **анимации палитр** (hue shifting) и генеративного дизайна
- Как визуальный инструмент редактирования кривых в advanced-режиме интерфейса

**Где Poline НЕ заменяет базовую кривую:**
- Ни одна из стандартных easing-функций Poline не воспроизводит распределение шагов Radix
- UI-кривая — это кусочная функция с разной крутизной на разных участках (медленный старт на 1–2, ускорение на 3–5, специфический пик на 9), синусоида или экспонента этого не даёт

**Техническая интеграция Poline:** MIT-лицензированная, zero-dependency, TypeScript. Нативно работает в HSL. Для OKLCH нужна адаптация (fork с заменой HSL на OKLCH-координаты, принцип полярных координат идентичен: H=угол, C=радиус, L=высота).

#### Итого: иерархия кривых

```
┌─────────────────────────────────────────────────┐
│  Уровень 0: Radix-extracted curve (дефолт)      │
│  → результат максимально близок к Radix          │
│  → проверен, знаком дизайнерам                   │
├─────────────────────────────────────────────────┤
│  Уровень 1: Модифицированная базовая кривая     │
│  → тот же принцип, но с ручной коррекцией       │
│  → пользователь двигает точки на кривой          │
├─────────────────────────────────────────────────┤
│  Уровень 2: Poline presets (advanced)           │
│  → sinusoidal, exponential, cubic и т.д.         │
│  → для нестандартных палитр и гармоник            │
│  → не для стандартных UI-шкал                     │
└─────────────────────────────────────────────────┘
```

### 3.10. Модуль цветовых гармоник (Color Harmony Engine)

Отдельный функционал — генерация НЕ-UI палитр по кругу Иттена:

**Поддерживаемые схемы:**
- Комплементарная (180°)
- Сплит-комплементарная (150° + 210°)
- Триадная (120° + 240°)
- Тетрадная (90° + 180° + 270°)
- Аналоговая (±30°)
- Квадратная (90° + 180° + 270°)

**Преимущество OKLCH:** При генерации гармоник достаточно менять H (hue), сохраняя L и C — получаем цвета с одинаковой воспринимаемой яркостью и насыщенностью. В HSL это невозможно.

### 3.10. Модуль экспорта (Export Layer)

**Приоритет форматов:**

**P0: SVG-матрица** — основной формат. Визуальная сетка всех цветов с подписями `{role}-{step}` (например `brand-01`, `brand-09`, `success-01`, `neutral-12`). Solid + alpha, light + dark. Дизайнер вставляет в Figma или использует как референс.

```
brand-01  brand-02  brand-03  ... brand-12
brand-a01 brand-a02 brand-a03 ... brand-a12  (alpha)
success-01 success-02 ...
neutral-01 neutral-02 ...
```

**P1: CSS Custom Properties**
```css
--brand-01: oklch(0.99 0.005 262);
--brand-09: oklch(0.55 0.24 262);
--brand-a01: oklch(0.55 0.24 262 / 0.02);
/* HEX fallback */
--brand-09: #4A6CF7;
--brand-09: oklch(0.55 0.24 262);
```

**P2:** JSON, Tailwind config, Figma variables, Design tokens (W3C), SCSS/SASS

## 4. Нестандартные сценарии и решения

### 4.1. Пастельный бренд-цвет

**Проблема:** Если бренд-цвет — пастельный (например, L=0.85, C=0.08), то стандартный шаг 9 должен быть насыщенным. Но бренд-цвет — пастельный.

**Решения (переключаемые):**

| Стратегия | Поведение | Когда использовать |
|-----------|-----------|-------------------|
| `pastel-as-brand` | Шаг 9 = пастельный цвет бренда как есть, шкала строится от него | Бренд строго фиксирован, менять нельзя |
| `saturate-to-9` | Шаг 9 получает hue бренда с повышенной C, бренд-цвет сдвигается на шаг 4–5 | Нужна стандартная UI-палитра |

### 4.2. Жёлтый и его ограничения

**Проблема:** В OKLCH максимальная хроматика жёлтого (H≈100°) в sRGB-гамуте — около C=0.21 при L=0.90. При L=0.55 (стандартный шаг 9) хроматика может быть выше, но жёлтый при L=0.55 визуально выглядит "грязным" (оливковым).

**Решение:** Для hue в диапазоне 70°–110° (жёлто-оранжевый) шаг 9 получает повышенную L (≈0.80–0.85) с пометкой, что foreground-текст — тёмный. Именно так делает Radix (yellow, amber, lime, mint, sky — все с тёмным текстом на шаге 9).

### 4.3. Фиксированные цвета из брендбука

**Проблема:** Клиент даёт HEX бренд-цвета, который менять нельзя. Но генерация шкалы вокруг него создаёт отклонения.

**Решение:** Режим `fixed-brand`:
1. Шаги 9 и 10 — строго HEX из брендбука. Это не подлежит обсуждению.
2. Остальные шаги строятся от hue бренд-цвета по базовой кривой L/C
3. Переключатель: "шаги 9–10 строго от бренда" vs "допустимый сдвиг как Radix" (Radix слегка корректирует шаг 9 для лучшей гармонии шкалы — некоторым проектам это подходит, некоторым нет)
4. Фиксация касается только brand-цвета. Семантические цвета (success, warning, danger, info) генерируются свободно, их шаги 9–10 не привязаны к брендбуку

### 4.4. Один бренд — несколько акцентных цветов

**Проблема:** У бренда есть primary (синий) и secondary (зелёный). Оба нужны как полноценные 12-шаговые шкалы.

**Решение:** Множественный ввод brand-цветов. Semantic Hue Resolver учитывает все бренд-цвета при подборе семантических, избегая конфликтов (если бренд = зелёный, то success сдвигается к teal/jade).

## 5. Техническое ядро: почему OKLCH

### 5.1. Перцептуальная равномерность

В HSL lightness=50% для синего (H=240°) выглядит значительно темнее, чем L=50% для жёлтого (H=60°). В OKLCH L=0.5 — это одинаковая воспринимаемая яркость для любого hue. Это фундамент: когда мы строим шкалу из 12 шагов, каждый шаг на L-кривой равномерно отличается от соседей визуально, а не только математически.

### 5.2. Ортогональность осей

В OKLCH изменение H не влияет на воспринимаемую L или C. Изменение C не сдвигает hue (в отличие от CIE LCH, где при снижении chroma синий "уплывает" в фиолетовый). Это критично для программной генерации палитр — одно изменение даёт предсказуемый результат.

### 5.3. Wide-gamut готовность

OKLCH может кодировать цвета за пределами sRGB (Display P3 и шире). На устройствах Apple и OLED-экранах — +30% цветов. Генератор может предлагать P3-версии с sRGB-fallback.

### 5.4. Нативная поддержка в CSS

`oklch()` поддерживается в Chrome, Safari, Firefox. Можно экспортировать палитру сразу в OKLCH-нотации с HEX-fallback для старых браузеров.

### 5.5. APCA + OKLCH = предсказуемая доступность

Perceptual lightness OKLCH напрямую коррелирует с контрастом по APCA. Зная L фона и L текста, можно с высокой точностью предсказать Lc-значение контраста без полной конверсии в sRGB.

### Core (ядро) — zero-dependency UI

| Зависимость | Назначение |
|-------------|-----------|
| **Culori** | OKLCH/Oklab/P3 конвертация, интерполяция, гамут-маппинг |

### Figma Plugin

| Зависимость | Назначение |
|-------------|-----------|
| **React** + **Vite** + **vite-plugin-singlefile** | Сборка UI в один HTML (требование Figma) |
| **@figma/plugin-typings** | Типы Figma Plugin API |
| **core** (из monorepo) | Вся логика генерации |

Figma plugin состоит из двух частей:
- **UI** (`ui.tsx`) — React-интерфейс, рендерится в iframe. Импортирует core для расчётов. Упрощённая версия web-интерфейса, адаптированная под размеры окна плагина (ширина 320–400px)
- **Controller** (`controller.ts`) — взаимодействие с Figma API: создание Figma Color Styles, Figma Variables (для Design Tokens), применение цветов к выбранным фреймам, генерация SVG-матрицы прямо в Figma canvas

**Ограничение Figma plugin:** shadcn/ui нельзя использовать напрямую (нет Tailwind в iframe плагина). Варианты:
1. Написать легковесные стили для plugin UI вручную (CSS modules или inline styles)
2. Использовать `figma-plugin-ds` (компоненты в стиле Figma UI) — для нативного вида

### Math (общий для core)

- Все вычисления в OKLCH → конвертация в sRGB/HEX/P3 только на выходе
- Базовая кривая L/C — extracted из Radix (Cyan эталон)
- Гамут-маппинг: итеративное снижение C по алгоритму CSS Color 4 (binary search по chroma)

### 13.1. Основной сценарий (happy path)

**Контекст:** UI-дизайнер начинает проект. У него есть brand color из брендбука: `#2563EB` (синий). Нужна полная цветовая система для дизайн-системы.

**Шаг 1 — Ввод (3 секунды)**
Дизайнер вставляет HEX `#2563EB` в единственный инпут на главном экране. Нажимает "Generate" или Enter.

**Шаг 2 — Мгновенный результат (0.5 секунды)**
Система генерирует полную палитру:
- Brand: 12 шагов solid + 12 alpha
- Success (green): 12 + 12
- Warning (yellow/amber): 12 + 12
- Danger (red): 12 + 12
- Info (cyan/blue): 12 + 12
- Neutral (с лёгким тинтом бренда): 12 + 12

Всё показывается как **цветовая матрица** — таблица, где строки = роли, столбцы = шаги 1–12. Шаг 9 подсвечен — это "опорный" цвет.

**Шаг 3 — Проверка и настройка (10–30 секунд)**
Дизайнер видит:
- Контрастность текстовых пар (11 на 1, 12 на 1, 12 на bg) — зелёные/красные чекмарки

- Может подвинуть hue семантических цветов, если дефолтные не нравятся

**Шаг 4 — Экспорт (5 секунд)**

- SVG matrix → скачивается файл с матрицей и именами

**Итого: < 30 секунд от HEX до полной палитры.**

### 13.4. UI-принципы

**Progressive disclosure:** Главный экран — один инпут + матрица результата. Настройки — в боковой панели или аккордеоне. Advanced (кривые Poline, ручная коррекция шагов) — отдельная вкладка/модалка.

**Feedback loops:** Изменение любого параметра → мгновенная перегенерация матрицы (core работает <50ms). Контрастные проверки обновляются синхронно. Нарушения гамута подсвечиваются оранжевым.

**Responsive layout:** Desktop — матрица + sidebar рядом. Tablet — матрица сверху, настройки снизу. Не мобильный инструмент (дизайнеры работают на десктопе).

## 11. Метрики успеха

| Метрика | Цель |
|---------|------|
| Время создания палитры | < 30 секунд (vs 15–30 минут сейчас) |
| Количество ручных правок после генерации | < 3 (vs 10–20 сейчас) |
| Процент шагов, проходящих APCA Lc 60+ | > 95% для текстовых пар |
| Отклонение шага 9 от входного HEX | ΔE < 1 (незаметно глазу) |
| Визуальная равномерность шкалы (равные перцептуальные интервалы L) | Отклонение L между шагами < 5% от целевого |

## 12. Открытые вопросы — решения

| # | Вопрос | Решение |
|---|--------|---------|
| 1 | Форма кривой L/C для 12 шагов | Базовая = extracted Radix curve. Альтернативные формы — как переключаемые пресеты |
| 2 | Hue rotation в шкале | Переключаемое property: вкл/выкл. Где-то полезно для натуральности, где-то мешает |
| 3 | Адаптация к экранам | Не нужна. Только переключатель гамута sRGB / P3 |
| 4 | Эффект Хелмгольца-Кольрауша | Переключаемое property: вкл/выкл. Компенсация насыщенности для шагов 9–10 |
| 5 | Порог equal-chroma для всех hue | Для каждого шага (1–12) C корректируется индивидуально, чтобы все hue были доступны в рамках гамута. В основном корректируется C, редко L. Хроматика разная для разных hue — это нормально |
| 6 | Приоритет экспорта | **SVG-матрица** с именами `color-00`…`color-12` для каждого оттенка — первый приоритет. Затем CSS, JSON, Tailwind и т.д. |
